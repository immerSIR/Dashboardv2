# Backend API Reference — Map Action (Mapapi)

This documents the Django REST backend that powers this dashboard. Source repo: **https://github.com/223MapAction/Mapapi** (Django + DRF + SimpleJWT + Celery/Redis + Postgres/Supabase). Production base URL: **`https://api.map-action.com`** (set in `src/config/api_url_base.jsx`).

> Scope: everything the frontend needs — endpoints, auth, request/response schemas, enums, and gotchas. Generated by studying the backend source (`Mapapi/urls.py`, `views/`, `serializer.py`, `models.py`, `permissions.py`, `backend/settings.py`).

> ### 🔎 Interactive API docs (Swagger / OpenAPI)
> The backend ships **live, always-in-sync** interactive docs — the fastest way to explore and *try* endpoints:
> - **Swagger UI:** `<backend>/MapApi/schema/swagger-ui/` — click **Authorize**, paste your `access` token (from `POST /MapApi/login/`), and call any endpoint.
> - **ReDoc:** `<backend>/MapApi/schema/redoc/` · **Raw schema:** `<backend>/MapApi/api/schema/`
>
> On our deployment `<backend>` = `https://backend-production-0726b.up.railway.app`. Every operation is tagged by domain (Incidents, Collaboration, Organisations, Auth…) with real request/response bodies, params, and error codes. This Markdown file remains the curated companion (rationale, gotchas, discrepancies); Swagger is the exhaustive machine-generated reference.

> 🇫🇷 A French version exists: [`BACKEND_API.fr.md`](./BACKEND_API.fr.md). Keep both files in sync.

> **⚠️ IDs are UUIDs (2026).** Every primary key and foreign key is now a **UUID string** (e.g. `"3f9a…-…"`), not an integer — to prevent id enumeration (IDOR). Detail routes are `/<uuid>` (e.g. `GET /MapApi/incident/<uuid>`). On the frontend, **never `parseInt()` an id**; compare ids as strings. DB tables also moved from the `extensions` schema to **`public`**. The migration invalidated old sessions (tokens carried the old integer user id) — users re-login once.

## 1. Conventions (read first)

- **URL prefix:** every endpoint below is under **`/MapApi/`** (e.g. `https://api.map-action.com/MapApi/incident/`). Root routing: `backend/urls.py` → `path('MapApi/', include('Mapapi.urls'))`.
- **Trailing slashes are inconsistent and significant.** Some routes end with `/`, some don't (e.g. `incident/<id>` has NO slash; `incident/` does). Each endpoint below shows the exact path. Match it exactly — Django will not auto-redirect a POST.
- **Auth header:** `Authorization: Bearer <access_token>` (JWT). See §2.
- **⚠️ Most endpoints are NOT authenticated.** Many list/stat/reference endpoints declare `permission_classes = ()` (fully public). Only the newer incident-collaboration, organisation-member, profile, and trash/bulk endpoints require auth. The "Auth" line on each endpoint is authoritative. Do **not** assume the backend gates data — it frequently doesn't.
- **CORS:** `CORS_ALLOW_ALL_ORIGINS = True` — any origin may call the API.
- **Pagination:** list endpoints using `CustomPageNumberPagination` return `{ "count", "next", "previous", "results": [...] }`. Query params: `?page=<n>&page_size=<n>` (default `page_size=100`, max `1000`). A few older lists use DRF's default `PageNumberPagination` (page size 10, `?page=` only). Several endpoints return a **plain array** (no pagination) — noted per endpoint. **Frontend code already handles both shapes** (`data.results ?? data`); keep doing that.
- **Content types:** JSON for most writes. Endpoints that accept files (`photo`, `video`, `audio`, `attachment`, `proof_image`, `proof_video`, `file`, `logo`) require **`multipart/form-data`**.
- **Media URLs:** uploaded files are served under `/uploads/...` (`MEDIA_URL`); image/video/voice/document fields may also be backed by Supabase storage. Treat file fields as URLs returned by the API.
- **Error shape:** not standardized. Validation errors → DRF dict `{ "field": ["msg"] }` (400). Custom errors → `{ "error": "message en français" }` or `{ "detail": "..." }`. Some 404s have empty bodies. Status codes are sometimes non-idiomatic (e.g. `GET /image/` returns `201`). Don't rely on status alone — inspect the body.
- **Language:** error/notification messages are in **French**.

## 2. Authentication

JWT via `rest_framework_simplejwt`. Configured in `backend/settings.py`:

- **Access token lifetime: 90 days.** Refresh token: 1 day. Tokens are **not** rotated/blacklisted. (This long-lived access token is why the frontend can persist a session in `sessionStorage` for a long time.)
- `USERNAME_FIELD = 'email'` → **login uses `email`, not `username`.**
- Default auth class is `CookieJWTAuthentication` (**Bearer-first**, then httpOnly-cookie fallback); there is **no** global `DEFAULT_PERMISSION_CLASSES`, so a view without `permission_classes` is effectively **public**.
- **The dashboard authenticates with `Authorization: Bearer` (token in `sessionStorage`), not cookies.** httpOnly cookies were tried but can't work here: the frontend and backend live on different sites (`*.up.railway.app` is a public suffix, and local dev runs on `localhost`), so the auth cookie is third-party and browsers block it. The login response returns `access`/`refresh` in the body for exactly this reason. Don't reintroduce cookie-only auth unless the FE and BE are served same-site (shared registrable domain) or behind a same-origin proxy.

### Primary login flow (used by the dashboard)

| Step | Endpoint | Returns |
|---|---|---|
| 1. Obtain tokens | `POST /MapApi/login/` (alias `POST /MapApi/api/token/`) | `{ "access", "refresh" }` |
| 2. Fetch current user | `GET /MapApi/user_retrieve/` (Bearer) | `{ "status", "message", "data": <User> }` |
| 3. Refresh | `POST /MapApi/token/refresh/` `{ "refresh" }` | `{ "access" }` |
| 4. Verify | `POST /MapApi/verify-token/` `{ "token" }` | `200 {}` or `401` |

`POST /MapApi/login/` body: `{ "email", "password" }`. On bad credentials → `401`.

`GET /MapApi/user_retrieve/` is the canonical "who am I" call. Response `data` is a `UserSerializer` object (see §4) including the read-only `organisation_name`. (The frontend reads `response.data.data || response.data`.)

### Other auth-related endpoints

| Method · Path | Auth | Body → Response |
|---|---|---|
| `POST /MapApi/register/` | none | `{first_name,last_name,email,password,phone,address}` → `201 { user, token:{refresh,access} }` (auto-login on register). `GET` lists all users. |
| `POST /MapApi/registerCitizen/` | none | `{ email }` → `201` sends email-verification link. |
| `GET /MapApi/verify-email/<uuid:token>/` | none | marks `is_verified=true`. |
| `PUT/PATCH /MapApi/set-password/` | Bearer | `{ password }` → sets password for current user (post-verification). |
| `POST /MapApi/gettoken_bymail/` | none | `{ email }` → `201 { token }` (raw **access** token, **no password check** — issues a token from an email alone; treat as sensitive). |
| `GET /MapApi/get_csrf_token/` | none | `{ csrf_token }`. |

### Password reset / change (matches `authService.js`)

| Method · Path | Auth | Body → Behavior |
|---|---|---|
| `POST /MapApi/password/` | none | `{ email }` → emails a 7-char alphanumeric `code` (valid ~1h). `201 {status,message}`. |
| `POST /MapApi/password_reset/` | none | `{ email, code, new_password, new_password_confirm }` → resets password. Errors return `{status:"failure", message}` with messages like `"non matching passwords"`, `"reset code has expired"`, `"no such item"`. |
| `PUT/PATCH /MapApi/change_password/` | Bearer | `{ old_password, new_password }` → `200 {status,code,message,data}`; wrong old password → `400 {"old_password":["Wrong password."]}`. |

> ⚠️ The frontend `authService.changePassword` sends `{ old_password, new_password }` via `POST`. The backend `ChangePasswordView` is an `UpdateAPIView` (expects `PUT`/`PATCH`). Verify the method — a `POST` may 405.

### Field-agent authentication (mobile/field, not the web dashboard)

Field agents (`org_role = field_agent`) don't use email/password. They log in with a generated **`agent_code`** or **phone + 4-digit PIN**:

- `POST /MapApi/agent-login/` `{ agent_code }` → `{ access, refresh, user:{...} }`. Errors: `401 "Code agent invalide."`, `403 "Ce compte est désactivé."`.
- `POST /MapApi/agent-pin-login/` `{ phone, pin }` → `{ access, refresh, user:{..., must_change_pin } }`. If `must_change_pin` is true, force a PIN change.
- `POST /MapApi/agent/change-pin/` (Bearer, field agents only) `{ old_pin, new_pin }`. `new_pin` must be 4 digits and not trivial (`0000/1234/1111/...`).

### OTP / phone (SMS via Orange Mali)

- `POST /MapApi/otpRequest/` `{ phone }` → sends 6-digit OTP (5-min expiry).
- `POST /MapApi/verifyOtp/` `{ phone, otp }` → `{ access, refresh, user }`.
- `GET|POST /MapApi/verify_otp/` — `PhoneOTP` TOTP variant (field `phone_number`).

## 4. Core schemas

Field names are **verbatim** from the models/serializers. `auto` = server-set. Foreign keys are returned as the related id unless a serializer nests the full object (noted). Note legacy misspellings used as-is: **`lattitude`** (incidents/zones), **`adress`**, **`colaboration`**.

### User (`UserSerializer` — returned by `user_retrieve`, nested in incidents)
Excludes `password`, `is_superuser`, `is_active`, `is_staff`, `user_permissions`. Key fields:
`id`, `email`, `first_name`, `last_name`, `phone`, `avatar` (image url, default `avatars/default.png`), `address`, `user_type` (enum), `organisation` (free-text legacy string), `organisation_member` (FK id → Organisation), **`organisation_name`** (read-only, from the linked org), `org_role` (enum or null), `agent_code`, `must_change_pin`, `points` (int), `zones` (M2M zone ids), `is_verified`, `provider`, `community` (FK), `date_joined` (auto), `is_deleted`.
Write-only on create: `password`, `incident_preferences` (list, for élus).
> Two org concepts coexist: `organisation` (legacy free string) and `organisation_member` (real FK to `Organisation`). Prefer `organisation_member` / `organisation_name`.

### Organisation (`OrganisationSerializer`, `fields=__all__`)
`id`, `name` (unique), `acronym`, `subdomain` (unique), `is_premium`, `logo` (image url), `organisation_type` (enum), `activity_sector` (enum), `intervention_country` (enum), `partner_status` (enum, default `active`), `description`, `phone`, `website_url`, `primary_color`/`secondary_color`/`background_color` (hex strings), `created_at` (auto), **`members_count`** + **`incidents_taken_count`** (read-only — incidents whose leader `taken_by` belongs to the org). `OrganisationMemberSerializer` (used by member endpoints **and `/agents/`**) exposes a user as: `id, email, first_name, last_name, phone, **avatar** (image URL), organisation_member, organisation_name, org_role, web_role, agent_code, is_active, date_joined`.

### Incident (`IncidentSerializer` write / `IncidentGetSerializer` read)
`id`, `title`, **`zone`** (string, **required**), `description`, `photo` (image), **`thumbnail`** (image URL — **auto-generated** ~320px from `photo` on save, read-only; use it for the incidents tab instead of the full `photo`), `video` (file), `audio` (file), `lattitude` (string), `longitude` (string), `etat` (enum, default `declared`), `user_id` (FK → reporter User), `category_id` (FK → Category), `indicateur_id` (FK → Indicateur), `category_ids` (M2M Category), `slug`, `taken_by` (FK → leader User, nullable), `take_in_charge_mode` (enum/null), `resolution_start_date`, `resolution_end_date` (dates; both required to close), **`progress`** (0–100, read-only, auto from confirmed tasks), `is_public` (default true), `is_deleted` (soft-delete flag), `created_at` (auto). Read-only computed: `reported_by_agent` (bool — reporter is a field agent).
- **`IncidentGetSerializer`** nests the full `user_id` (User) and `category_id` (Category) objects; used by most list endpoints.
- The OneToOne AI `prediction` is accessed via its own endpoints (see §6.7), not embedded here.
- **Acting organisations (list + detail)** *(2026)* — both serializers now expose, so another org sees who is already working on an incident **before** opening it:
  - **`taken_by_organisation`**: `{id, name}` of the org that took the incident in charge, or `null`.
  - **`taken_by_name`**: full name of the person who took charge (not just the `taken_by` id).
  - **`acting_organisations`**: `[{id, name, relation}]` — every org active on the incident; `relation` ∈ `leader` (took charge) · `assigned` (super-admin org assignment accepted) · `collaborator` (collaboration accepted). Deduplicated, strongest relation wins. (Only **accepted** collaborations appear here.)
  - **`my_collaboration`** *(2026)*: the **viewer's own** collaboration request on this incident (or their org's), **whatever its status** — `{id, status(pending|accepted|declined), role, created_at, organisation_id, organisation_name}`, or `null` if none. Lets the list show *"j'ai demandé à collaborer — en attente"* on an incident already led by another org. **Only populated on `GET /incident/`** (the main list, which passes the request context); `null` on endpoints without viewer context.
- **`org_assignments`**: `[{id, organisation_id, organisation_name, status, deadline}]` (pending/accepted/declined super-admin assignments).

### Collaboration (`CollaborationSerializer` / `CollaborationEnrichedSerializer`)
`id`, `incident` (FK), `user` (FK — the collaborating user/org rep), `role` (enum), `status` (enum, read-only — set via accept/decline), `motivation`, `end_date` (must be future), `other_option`, `created_at` (auto). Unique together `(incident, user)`.
Read-only enrichments: `organisation_name`, `organisation_id`, `user_full_name`, `user_email`, `incident_title`, **`incident_photo`** *(2026)*, **`incident_thumbnail`** *(2026)* (incident image URLs, for the collaboration cards), `incident_details` (full Incident), `prediction_details` (full Prediction). The dashboard serializer adds `user_role`, `incident_description/zone/etat/progress`, **`incident_photo`/`incident_thumbnail`**, `start_date` (=created_at), `participants_count`.
- **`sender` / `receiver`** *(2026)* — explicit parties of a collaboration request, both `{id, name, email, organisation_id, organisation_name}`: **`sender`** (émetteur) = the org requesting to join (`collaboration.user`); **`receiver`** (récepteur) = the leader who receives the request (`incident.taken_by`), or `null` if none yet. Use these in the "demande" tab so the logged-in org can tell whether it sent or received a request (don't infer from `user` alone).

### IncidentTask (`IncidentTaskSerializer`, `fields=__all__`)
`id`, `incident` (**read-only — injected from the URL; do NOT send it in the body**), `title` (required), `description`, `start_date`, `end_date` (required; start ≤ end), `state` (enum), `proof_image`, `proof_video` (required when `state=done`), `failure_reason` (required when `state=failed`), `assigned_to` (FK User — **optional**; omit for an unassigned task, or send a **valid agent UUID** — never a legacy int id), `created_by` (auto = current user), **`is_confirmed`** (read-only; auto-true if leader created it; only confirmed tasks count toward incident `progress`), `created_at`/`updated_at` (auto).

### PartnerSuggestion (`PartnerSuggestionSerializer`)
`id`, `incident` (FK), `suggested_by` (auto = current user), `suggested_partner` (FK User — optional if `suggested_organisation` sent), `suggested_organisation` (write-only org id → resolves to that org's admin/bureau user), `suggested_role` (`contributor`|`observer`), `justification` (required), `status` (read-only enum). Read-only names: `incident_title`, `suggested_by_name`, `suggested_by_organisation`, `suggested_partner_name`, `suggested_partner_organisation`. Unique `(incident, suggested_partner)`.

### IncidentAssignment (`IncidentAssignmentSerializer`)
`id`, `incident` (FK, from URL), `agent` (FK User — must be `field_agent` in the incident's org), `assigned_by` (auto), **`deadline`** (datetime, **required**), `status` (enum), `created_at`/`updated_at`. Read-only: `agent_name/email/phone`, `incident_title`, `assigned_by_name/email`, `incident_detail` (full incident). Unique `(incident, agent)`.

### FieldReport (`FieldReportSerializer`)
`id`, `agent` (auto), `incident` (auto from body `incident`/`incident_id`), `location_lat`, `location_lon` (strings — must be within 100 m of incident), `distance_meters` (auto), `notes`, `photo` (image), `visited_at`, `created_at`. Read-only: `agent_name`, `incident_title`, `incident_zone`.

### DiscussionMessage (`DiscussionMessageSerializer`)
`id`, `incident` (auto), `collaboration` (auto), `sender` (nested User, read-only), `message` (text), `audio` (file), `attachment` (pdf/doc/docx/xls/xlsx only), `recipient` (nested User, optional), `created_at`. At least one of `message`/`audio`/`attachment` required.

### Prediction (`PredictionSerializer`, `fields=__all__`, almost all read-only)
The AI/geo/impact analysis for an incident (OneToOne `incident`). Large object — key fields a frontend would render:
- Status/meta: `id`, `prediction_id`, `incident` (FK), `status` (enum), `error_message`, `created_at`, `updated_at`.
- Classification: `incident_type`, `macro_category`, `sub_category`, `description`, `analysis`, `piste_solution`, `recommendation`.
- Severity/impact: `global_impact_score` (float), `base_severity` (int), `impact_tags` (list), `impact_radius_meters`, `radius_explanation`, `source_size_meters`, `spread_vectors` (list).
- Geo: `latitude`, `longitude`, `city`, `region`, `country`, `display_name`.
- Social/human exposure: `social_vulnerability_score`, `total_population_exposed`, `adult_men_exposed`, `adult_women_exposed`, `children_exposed`, `health_centers`, `maternities`, `schools`, `nurseries`, `markets`, `water_points`, `main_roads_bridges`, `residential_buildings`.
- Raw JSON blobs: `ai_analysis`, `topography`, `satellite`, `social_data`, `human_impact`, `geocoding`, `potential_risk`, `full_response`.
- Legacy plot fields: `ndvi_heatmap`, `ndvi_ndwi_plot`, `landcover_plot`, `legacy_incident_id`.

### Other reference models (all `fields=__all__`)
- **Zone:** `id, name` (unique), `description`, `lattitude`, `longitude`, `photo`, `created_at`.
- **Category:** `id, name` (unique), `description`, `photo`, `created_at`.
- **Indicateur:** `id, name` (unique), `created_at`.
- **Rapport (report):** `id, details, type` (`"zone"` for zone-wide), `incident` (FK), `zone` (string), `user_id` (FK requester), `date_livraison`, `statut` (enum), `incidents` (M2M), `disponible` (bool), `file`, `created_at`.
- **Message:** `id, objet, message, zone` (FK), `communaute` (FK), `user_id` (FK target élu), `created_at`. **ResponseMessage:** `id, response, message` (FK), `elu` (FK), `created_at`.
- **Communaute:** `id, name, zone` (FK), `created_at`.
- **Contact:** `id, objet, message, email, created_at`.
- **Evenement (event):** `id, title, zone, description, photo, date, lieu, video, audio, user_id` (FK organizer), `latitude`, `longitude`, `created_at`.
- **Participate:** `id, evenement_id` (FK), `user_id` (FK), `created_at`.
- **Notification:** `id, user` (FK), `message`, `read` (bool), `colaboration` (FK Collaboration — note spelling), `created_at`.
- **UserAction:** `id, user` (FK), `action`, `timeStamp`.
- **ChatHistory:** per-message `{incident, user, role, content, created_at}` (the incident AI chat, scoped per user via `incidents/<id>/chat/`). The old `{session_id, question, answer}` columns are deprecated/unused — their endpoints were removed.

## 5. Roles & permissions model

The collaboration permissions (`permissions.py`) resolve the **leader** of an incident as: a user with an `accepted` `Collaboration(role='leader')`, **or** `incident.taken_by == user`. Custom classes:

| Permission | Allows |
|---|---|
| `IsIncidentLeader` | leader only |
| `IsIncidentCollaborator` | any `accepted` collaborator (leader/contributor/observer) or `taken_by` |
| `IsIncidentContributor` | accepted `contributor` (read open to all collaborators) |
| `IsIncidentLeaderOrContributor` | leader or accepted contributor (used for tasks/suggestions create) |
| `IsIncidentLeaderOrReadOnlyCollaborator` | read = any collaborator; write = leader |
| `IsSuperAdmin` | `is_superuser` (trash, bulk-restore, force-delete, prediction retry) |
| `IsSuperAdminOrOrgOwnIncident` | super admin → any incident; org member → only incidents reported/taken by their org (used for delete) |

Organisation-member endpoints additionally gate on `org_role`: `org_admin` and `bureau_agent` manage members; only `org_admin` creates staff; `is_staff` bypasses org checks.

## 7. Frontend integration notes

How this maps to the existing `dashboard/` code, plus discrepancies worth verifying:

1. **Auth is already wired correctly.** `authService.login` → `POST /MapApi/login/` then `GET /MapApi/user_retrieve/`; refresh → `POST /MapApi/token/refresh/`; password flows match §2. The 90-day access token explains long-lived sessions.
2. **`createAuthenticatedAxios()` is right** — `Authorization: Bearer <access>` is exactly what the protected endpoints expect.
3. **Incident update — likely wrong URL.** `updateIncidentService` does `PUT /MapApi/incidents/<id>/` (plural, trailing slash). The backend's incident update is `PUT /MapApi/incident/<id>` (**singular, no slash**). The plural route doesn't exist → expect `404`. Use `incident/<id>` and include an `etat` field (the view reads `request.data['etat']`).
4. **Agent assignment — verify the endpoint.** `assignIncidentToAgentService` POSTs to `/MapApi/agent/assigned-incidents/`, but that route is **GET-only** (lists a field agent's assignments). To **create** an assignment, POST to `/MapApi/incidents/<id>/assignments/` with `{agent, deadline}`. The GET form is correct for listing.
5. **`changePassword` method.** Frontend uses `POST`; backend `change_password` is an `UpdateAPIView` (`PUT`/`PATCH`). Confirm which the deployed server accepts.
6. **Incident create** must be `multipart/form-data` with **`zone` required** (already handled via `FormData` in `createIncidentService`).
7. **`etat` values** (`declared`, `taken_into_account`, `in_progress`, `resolved`) match the frontend's label/color maps in `incident_service.jsx`. Note backend also exposes `take_in_charge_mode` (internal/collaborative) and a server-computed `progress` (0–100) you can display directly.
8. **Notifications** (`getNotifications` in the header) only returns notifications tied to a **pending** collaboration — it's effectively a "pending collaboration requests" feed, not a general notification center.
9. **Two "organisation" fields** exist on a user: the legacy free-text `organisation` string and the real `organisation_member` FK (+ read-only `organisation_name`). The frontend currently stores `userData.organisation`; prefer `organisation_member`/`organisation_name` for org features.
10. **Public endpoints leak data.** Reference data, user lists, and most incident stats need no token. Don't rely on the API to hide anything; gate sensitive views in the frontend and treat `gettoken_bymail/` (token from email alone) and the hardcoded JWT signing key in the backend as known security weaknesses.
11. **"Inviter des organisations" sends `suggested_organisation`, not `suggested_partner`** *(fixed 2026-06)*. The invite UI picks an **organisation**, so the suggestion POST must send `suggested_organisation: <orgId>` (the backend resolves it to that org's admin/bureau). Sending the org id as `suggested_partner` (a **User** FK) yields `Invalid pk`. Also note: `PartnerSuggestion` has `unique_together (incident, suggested_partner)`, so `PartnerSuggestionSerializer` overrides `get_unique_together_validators()` to `[]` and checks uniqueness in `validate()` — otherwise DRF's auto `UniqueTogetherValidator` forces `suggested_partner` required and blocks the `suggested_organisation` path. Frontend: `suggestCollaborationPartnerService` + `IncidentDetail.handleJoinSubmit` + `CollaborationDetail`.
